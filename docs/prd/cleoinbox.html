Cleoinbox — PRD

The single spec we build from.

1. One-liner

A local desktop app with a chat sidebar that manages your inbox by plain-English command — "hide all my Uber receipts and never tell me about them again" — by compiling that intent into a native Gmail filter that Google enforces server-side, forever. No backend, no database.

2. Decisions log (ADR-style — why this shape)

  • The AI runs once, locally, at rule-creation time. It translates intent into a Gmail filter spec and writes it via the Gmail API. After that, Google enforces the filter on its own servers — machine off or on, no compute of ours, no email stored anywhere by us. The app is a thin local client over the provider's own rule engine. We do not build an email-processing backend.
  • Not Mail.app / AppleScript — Gmail API, server-side filters.
  • Multi-account.
  • Electron, not Tauri.
  • Monorepo (pnpm workspace). We may add a server later; retrofitting separate repos into a monorepo is painful, so set it up as a monorepo from the start.
  • No server, no DB in v1.
  • Electron setup locked early (irreversible bits): getting it wrong is easy, and the cost is asymmetric — a bad install means either eternal bug-chasing or a catastrophic break (appId, user-data paths, keychain-encrypted secrets, and security flags can't be safely changed after release). If setup misbehaves, assume it was installed wrong. Find a reputable setup guide, vet where it comes from, confirm it's current as of 2026, and follow it religiously.
  • Outlook is out of v1
  • Open source.

3. Constraints

  • No backend service; no database; nothing of ours runs continuously (v1).
  • Multi-account: N Gmail accounts, each independently authorized.
  • Bring-your-own-key: user supplies their own LLM key and their own Google OAuth client.
  • Free Gmail API quota only (personal-scale).
  • Google restricted-scope reality (gmail.modify, gmail.settings.basic, gmail.readonly): public distribution needs Google's CASA assessment. v1 lives in "testing" mode (≤100 users, 7-day refresh tokens) — no verification, no cost; CASA is a phase-2 gate.
  • The repo is guarded by opal-source (opal init) — no secrets or personal data reach the public remote.

5. The rule model

  • New mail: a command becomes a FilterSpec, written to Gmail via createFilter. Google then applies that filter to every email that arrives afterward.
  • Existing mail: filters never touch mail you already have, so a "hide X" command also runs applyToBacklog over current matches (archive / skip-inbox only in v1).
  • "Hide" semantics: default = archive / skip-inbox (still searchable). Trash (explicit "delete") is phase 2.

6. Safety

  • Destructive bulk ops (bulk trash/delete) are out of v1 — v1 does bulk archive only. When they land (phase 2) they require explicit chat confirmation showing the match count.
  • opal guards the repo; CLAUDE.md secret/PII rules apply.
  • Desktop OAuth client secret is not truly secret (PKCE); still never committed (.env gitignored, keychain at runtime).

7. v1 build order

  1. opal-source guard. opal init; confirm no secrets or personal data can reach the public remote before any code lands — work safe from step one.
  2. Electron. Install and boot Electron from a vetted, 2026-current guide (setup is irreversible — see Decisions).
  3. Monorepo skeleton. pnpm workspace; shared types; server stub; keychain helper.
  4. Gmail auth + read. PKCE OAuth; list inbox; multi-account.
  5. Chat → filter. Chat compiles one intent; createFilter; verify in Gmail.
  6. Backlog + bulk. applyToBacklog; bulk archive (skip-inbox) only.
  7. Rule management (read back filters.list, undo where possible).

8. Phase 2 (post-v1, out of scope)

  • Outlook, HostedProvider, hosted tier, Google CASA verification.
  • Destructive bulk ops (bulk trash/delete of existing mail).

9. Tech stack

Electron (Node main + React/TS renderer) · pnpm workspace monorepo · Gmail API · BYO LLM key · OS keychain (safeStorage) · opal-source (repo guard). No Rust.
